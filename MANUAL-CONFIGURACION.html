Añadir rutas y entender cada campo

Primero, la guía rápida para quien gestiona las rutas del día a día (sin saber Git ni programación). Después, campo a campo, qué hace cada valor de rutas/config.json y rutas/theme.json, con una miniatura de la app señalando exactamente dónde aparece.

Guía rápida — sin Git ni código

Todo esto se hace desde el navegador, en la página de GitHub. No hace falta instalar nada ni usar la terminal.

Antes de empezar

  • Acceso a GitHub con permiso para subir archivos a este repositorio (si no lo tienes, pídeselo a quien mantiene la web — te tiene que añadir como colaborador).
  • El archivo .gpx de la ruta, exportado desde donde la hayáis grabado (Wikiloc, Strava, GPS del club, gpx.studio...).
  • Saber de qué tipo es la ruta (MTB, ROAD, GRAVEL, TRAIL, BIKEPACKING, CASTELLBIKE...) — o crea uno nuevo si no encaja ninguno, ver más abajo.

Añadir una ruta nueva

  1. 1Entra en github.com/vgonzcam/ownGpxRutasApp, carpeta rutas → carpeta del tipo de tu ruta (p. ej. ROAD o GRAVEL).
  2. 2Botón "Add file" → "Upload files" → arrastra tu .gpx.
  3. 3Baja al final, escribe un mensaje breve y haz clic en "Commit changes".
  4. 4Espera 1-2 minutos. Puedes comprobar el progreso en la pestaña "Actions" del repositorio (check verde ✅ = terminado).
  5. 5Abre la web y busca tu ruta. Si no aparece, recarga forzado (Ctrl+Shift+R) o pruébalo en una ventana de incógnito.

Añadir un tipo de ruta completamente nuevo

Los tipos de ruta no son una lista cerrada en el código: son, literalmente, las carpetas que existen dentro de rutas/. Si el club empieza una modalidad nueva y ninguna de las carpetas actuales encaja, créala tú mismo:

  1. 1Carpeta rutas → "Add file" → "Upload files".
  2. 2Arrastra tu primer .gpx de ese tipo. Justo encima de la zona de subida hay un campo con la ruta de destino: escribe delante del nombre el de la carpeta nueva, por ejemplo XC/mi-primera-ruta-xc.gpx.
  3. 3"Commit changes" — GitHub crea la carpeta sola, en el mismo paso, porque no existía.

En el siguiente despliegue, el filtro nuevo (p. ej. "XC") aparece solo en el listado, al final de la fila. Si quieres controlar su posición, hay una lista opcional de orden en rutas/config.json → filters.trackTypes — ver la sección Filtros más abajo.

Cómo se calculan los datos de una ruta (y cómo corregirlos)

En cuanto subes el .gpx, la web calcula sola todo lo que se ve en el listado y el detalle, a partir de los puntos del track. Son cálculos automáticos y a veces no aciertan del todo — puedes corregir cualquiera de ellos sin tocar el archivo GPX.

Para corregir cualquiera de estos campos: abre rutas/config.json en GitHub, lápiz ✏️ para editar, y añade una entrada en routeOverrides con la clave exacta del archivo (carpeta + nombre, tal como aparece en manifest.json). Solo hace falta poner los campos que quieras corregir — el resto sigue calculándose solo.

Sintaxis exacta, ejemplo con los 11 campos a la vez y la miniatura de cómo se ve aplicado: sección routeOverrides, justo abajo.

Es un archivo JSON: cada llave { que abras se cierra con }, y cada campo lleva una coma después excepto el último de su bloque. Si algo se rompe (la web deja de cargar rutas), entra en la pestaña "History" del archivo en GitHub y vuelve a la versión anterior.

routeOverrides object

Corrige a mano cualquier dato calculado automáticamente para una ruta concreta: nombre, ubicación, dificultad, duración, distancia, desnivel, altitudes, TrailRank o tipo de recorrido. Se indexa por el mismo "file" que aparece en manifest.json. Todos los campos son opcionales — pon solo los que necesites.

Ejemplo con los 11 campos a la vez
"routeOverrides": {
  "ROAD/mi-ruta-nueva.gpx": {
    "name": "Ruta del Pantà",
    "location": "Castelldans",
    "difficulty": "dificil",
    "duration": "4h 15m",
    "distanceKm": 68.4,
    "elevGain": 1450,
    "elevLoss": 1450,
    "maxAlt": 512,
    "minAlt": 210,
    "trailRank": 62,
    "routeShape": "Circular"
  }
}

En el día a día casi nunca rellenas los 11 a la vez — normalmente uno o dos. Aquí van todos juntos solo para ver cada clave en su sitio.

filters object

rutas/theme.json

Paleta de colores de toda la app, en dos variantes (light / dark). Debe reflejar exactamente los valores por defecto ya escritos en index.html — si difieren, este archivo gana en tiempo de ejecución.

Cómo funciona

No son 40 colores sueltos: son 12 "tokens" con nombre (primary, accent, destructive...) que luego la app combina para pintar botones, chips, badges, bordes, etc. Cambiar un token cambia todos los sitios que lo usan a la vez.

Regla de contraste: cualquier par bg/texto debe llegar a 4.5:1 (WCAG AA) para texto normal. Antes de cambiar un token, comprueba el contraste del nuevo color contra su pareja — varias paletas pegadas en este proyecto han fallado aquí (rojo claro + texto blanco, azul claro + texto blanco) y hubo que oscurecerlas un paso.

rutas/manifest.json

Este no se edita nunca a mano: se regenera automáticamente en cada despliegue a partir de los archivos .gpx que haya dentro de rutas/<TIPO>/. Cualquier cambio manual se pierde en el siguiente push.

El tipo (ROAD, GRAVEL...) tampoco es una lista fija: es, literalmente, el nombre de cada carpeta que hay dentro de rutas/. Crear una carpeta nueva y subir un .gpx dentro basta para que aparezca un filtro nuevo en la web, sin tocar ningún archivo. Paso a paso en la sección Añadir un tipo de ruta completamente nuevo, al principio de esta página.
